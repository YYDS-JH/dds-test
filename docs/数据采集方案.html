无本体数据采集系统方案 人类穿戴传感器设备直接采集 · 5 硬件模块 · WiFi + 485 混合拓扑 · NTP 时间同步 · MCAP 数据合并

1. 系统概览

本系统由 5 个硬件设备组成，由人直接穿戴完成数据采集（无机器人本体）。设备分为三个逻辑采集节点：ego 头环、左手腕相机模组、右手腕相机模组。采集完成后数据通过 WiFi 局域网上传至服务器，按时间戳对齐合并为统一数据集。

3. 通讯架构

3.1 RS-485 总线（外骨骼 ↔ 手腕相机）

左右手腕各一条独立 RS-485 总线，手腕相机模组作为主机，外骨骼作为从机。

  • 物理层：RS-485 半双工，波特率 ≥ 921600 bps（外骨骼 8KB/s 远小于带宽上限）
  • 协议层：自定义二进制帧协议（见 附录 A），主机轮询、从机应答
  • 同步：外骨骼数据到达后，相机模组以自身系统时钟打时间戳（已通过 NTP 同步）

3.2 WiFi 局域网（头环 · 左腕 · 右腕 · 服务器）

三台采集节点和服务器处于同一 WiFi 局域网（建议 5GHz 频段，802.11ac 或以上）。

  • 控制指令：HTTP（服务器 → 采集节点）
  • 数据上传：HTTP PUT/POST（采集节点 → 服务器），MCAP 文件
  • 时间同步：NTP over UDP（采集节点 → NTP 服务器）
  • 带宽：9 MB/s 控制指令可忽略不计；数据上传为事后批量传输，不对采集过程带宽产生竞争

4.2 同步精度

  • WiFi 局域网环境下 NTP 典型精度：1–10 ms
  • 相机帧间隔 33 ms（30 fps），外骨骼采样间隔 10 ms（100 Hz），IMU 采样间隔 5 ms（200 Hz）
  • 结论：毫秒级 NTP 精度满足 帧级对齐需求，足够本次应用

5. 采集流程

5.1 采集生命周期

① 准备阶段

三台设备开机，自动连接 WiFi，chronyd 与 NTP 服务器完成同步（通常在启动后 1–2 分钟内达到稳定精度）。设备本地 HTTP server 启动，监听控制端口（如 8080）。

② 启动指令

服务器向三台设备分别 POST /start，Body 携带目标开始时间戳（服务器当前时间 + 50ms 延迟，吸收网络抖动）。三台设备收到后各自返回 200 OK 确认。

③ 采集进行中

到达指定时间戳后，三台设备同步开始记录。每台设备以自身 NTP 同步的系统时钟为每条传感器数据打时间戳，写入本地 MCAP 文件。服务器可随时通过 GET /status 查询运行状态和已写入数据量。

④ 停止指令

服务器向三台设备 POST /stop，同样携带目标停止时间戳。各设备按指定时间停止，关闭 MCAP 文件。

⑤ 数据上传

采集结束后，服务器向每台设备 POST /upload 或各设备主动推送 MCAP 文件到服务器指定路径。上传完毕返回 200。

⑥ 数据合并

服务器收集三个 MCAP 文件，按时间戳对齐所有 channel，产出统一数据集（详见第 6 节）。

5.2 本地存储与 MCAP 结构

6. 数据整合

6.1 服务器端合并流程

  1. 收集三个 .mcap 文件到服务器
  2. 打开所有 channel，读取每条消息的 log_time（NTP 同步后的时间戳）
  3. 以 时间戳为键，执行多文件归并排序（k-way merge），输出统一的时序数据集
  4. 合并后的数据集提供两种视图：
    • 原始流视图：所有 channel 按时间交错，保留完整原始数据
    • 对齐帧视图：以相机帧时间戳为基准（30 fps），将最近邻的外骨骼和 IMU 数据与之对齐

6.2 时间戳筛选

由于三台设备的启动和停止严格对齐在同一时间戳，合并时可直接以 采集窗口 [T_start, T_stop] 作为有效区间，不在区间内的数据视为无效并丢弃。

7. 控制协议

7.1 设备端 HTTP API

每台采集节点在端口 8080 上启动一个轻量 HTTP server，提供以下端点：

POST /start

启动采集。Body 为 JSON：{"start_time_ns": 1715600000050000000} — 目标开始时间戳（Unix 纳秒）。设备收到后设置定时器，到达时间戳后开始写入 MCAP。返回 200 OK 或 409 Conflict（已在采集）。

POST /stop

停止采集。Body 为 JSON：{"stop_time_ns": 1715601800050000000} — 目标停止时间戳。到达后关闭 MCAP 文件。返回 200 OK。

GET /status

查询运行状态。返回 JSON：{"recording": true, "elapsed_s": 127.3, "bytes_written": 1145000000, "channels": ["/headset/cam0", ...]}。

GET /download

下载本次采集的 MCAP 文件。服务器主动发起 GET 拉取，或在 /stop 完成后由设备 POST 推送到服务器指定路径。

7.2 可靠性与重试

服务器在发送 /start 或 /stop 时，如未收到 200 OK（超时 2 秒），执行重试：最多 3 次，间隔 200 ms。任一设备 3 次重试后仍未确认，本次采集取消并报告操作者。

8. 故障处理

故障场景影响应对
某设备未响应 /start 该设备未开始录制，数据缺失一个节点 重试 3 次后告警，操作者判断是否继续
采集期间 WiFi 断开 NTP 同步丢失，时钟逐渐漂移；HTTP 状态查询不可用 本地存储不受影响；采集结束后设备重新联网再上传
采集期间某设备掉电 该设备数据丢失（未刷盘部分） MCAP 定期刷盘（每 5 秒 fsync），减少丢失量
NTP 服务器故障 三台设备时钟开始漂移 chronyd fallback 到本地时钟（stratum 10）；采集前检查同步状态
存储空间不足 写入失败，数据截断 采集前通过 /status 检查可用空间并告警；采集期间空间不足时停止并保留已写入数据
数据上传失败 服务器缺少部分 MCAP 文件 本地保留 MCAP 副本，支持手动重新上传或通过 USB 备份导出
